Here is the thing, implementing time is hard. To give your web application a sense of timing is more complicated than geolocalizing it. It doesn't make sense. A lot of companies still rely on cron, the Unix utility for time-based job scheduling. It is simple to set up but it doesn't scale; it's error-prone; and it's hard to debug.


More sophisticated tech companies have replaced cron with their own solution: Airbnbâ€™s Chronos, Yelpâ€™s Tron, NextDoorâ€™s NDScheduler. That's great. It's distributed, more reliable, and easier to use. But at what cost? Are you okay deploying, maintaining, and depending on a stale library in your precious code base?


Crono is the best of both worlds, simple and reliable. See an example for more details.

Authentication

Requests to all endpoints have to be authenticated with a Authorization: Bearer {token} header. See the example below for more details. Email crono@gduverger.com to request a token.

Example 

$ curl http://www.crono.com{{ reverse_url('post_job') }} \
  -X POST \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "trigger": {
      "name": "crontab",
      "params": {
        "expression": "* * * * *"
      }
    },
    "task": {
      "name": "request",
      "params": {
        "method": "GET",
        "url": "http://gduverger.com/"
      }
    }'

HTTP/1.1 201 Created
